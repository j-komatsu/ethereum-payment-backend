## PM/PMO エージェントの導入

### なぜ追加したか

ブロックチェーン/Spring/セキュリティの3エージェントレビューは「コードの正しさ」を見る。
それとは別に「計画の正しさ（粒度・順序・スコープ）」を見る観点が欠けていた。

個人学習プロジェクトなのでチーム管理・進捗追跡は不要。
「計画の質」に絞ってフェーズ節目に実施するのが適切なタイミングと判断した。

### レビューの3軸

| 軸 | 具体的に見たこと |
|---|---|
| 計画の粒度・抜け漏れ | 前提として必要なステップが省略されていないか |
| 依存関係・順序の妥当性 | 先にやるべきことが後回しになっていないか |
| スコープのブレ・リスク | 実現困難なポイント・学習効果が薄いタスク |

### CLAUDE.md への組み込み

フェーズ完了時またはフェーズ着手前にPM/PMOレビューを実施する規約を追加した。
セッション終了時チェックリストにも追加済み。

## Phase 3 完了後の PM/PMO レビューで判明した主要事項

### 🔴 テスト環境戦略が計画に欠けていた

JPYC は Polygon mainnet のみにコントラクトが存在する。
Polygon Amoy testnet に公式テストコントラクトはない。
Phase 4 の Transfer イベント監視をどのチェーンで動作確認するか、計画に記載がなかった。

**判断:** Hardhat `--fork` による mainnet フォーク（選択肢A）を推奨として PLAN.md に明記した。

```bash
npx hardhat node --fork ${WEB3J_POLYGON_ENDPOINT}
```

これにより:
- 実際の JPYC コントラクトをリスクなしで使える
- 実際のブロック・イベントデータで動作確認できる
- ガス代・実JPYC のリスクなし

### 🔴 テスト基盤の整備が Phase 6 Day 28 に集中しすぎていた

PaymentService.confirmPayment / SiweService の署名検証など、
副作用が大きいコードのテストを「後でまとめて書く」スタイルだった。

**判断:** 各フェーズの実装日にテストを追加する形に変更した。
- Day 17（入金検知実装）→ confirmPayment 単体テスト同日追加
- Day 21（SIWE実装）→ SiweServiceTest 同日追加
- Day 28 → 残りのカバレッジ向上・統合テストに特化

### 🟡 Phase 4 の重要な設計前提が計画に記載なし

PLAN.md に「最後に処理したブロック番号を DB に保存」とあったが、エンティティ設計が省略されていた。

**追加した内容:**
- `PollerState` エンティティ設計（token PK, last_processed_block BIGINT, updated_at）
- `REQUIRED_CONFIRMATIONS = 20`（Polygon での推奨確認ブロック数）

### 🟡 Phase 5-F（フロントエンド）の開始タイミングが曖昧だった

タイムライン表に「Week 5 / Week 5-F」と並列で書かれていたが、
SIWE バックエンドが完成していない状態でフロントの SIWE 連携は実装できない。

**判断:** Phase 5 バックエンド完了後に Phase 5-F を開始する「直列構成」に明記した。

### 🟡 EIP-712 の実装難易度が計画上で過小評価されていた

「Day 22 の1日でバックエンドが EIP-712 署名データを生成」と書かれていたが、
Web3j での domain_separator + structHash の手動実装は格闘を伴う。

**判断:** Day 22-23 を「Permit + transferFrom」の2日構成として一体設計することを明記した。

## 学んだこと：PM/PMO の観点が補う盲点

コードレビューは「書いたコードの正しさ」を見る。
PM/PMO レビューは「計画の正しさ」を見る。

コードレビューでは発見されにくい問題:
- 「チェーンが違う」（Sepolia vs Polygon Amoy vs Polygon mainnet）
- 「テストを後でまとめて書くのはリスク」
- 「フロントとバックエンドは並行できない（依存がある）」
- 「1日で実装できない難易度のタスクがある」

これらは「実装してから気づく」と手戻りが大きい。
フェーズ境目で先に洗い出すことで、Phase 4 以降のスムーズな実装につながる。
