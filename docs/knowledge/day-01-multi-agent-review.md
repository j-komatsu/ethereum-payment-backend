# Day 01：マルチエージェントレビューの実際の価値

**日付:** 2026-05-24
**フェーズ:** Phase 0（初期構成）

---

## 何が起きたか

JPYC のコントラクトアドレスをコードに追加した際、末尾を誤って記述してしまった。

```java
// 誤り（私が設定した値）
JPYC("0x431D5dfF03120AFA4bDf332c61A6e1766eF37BF9", 18)

// 正しい値（レビューエージェントが検出）
JPYC("0x431D5dfF03120AFA4bDf332c61A6e1766eF37BDB", 18)
```

末尾3文字 `BF9` → `BDB` の違い。目視では見つけにくいレベルの誤り。

---

## なぜ重要だったか

Ethereum のアドレスは 42 文字の16進数。1文字でも違えば**全く別のアドレス**になる。

もしこの誤ったアドレスで本番稼働した場合：
- ユーザーが JPYC を送金しても、システム側の「監視アドレス」が間違っているため**入金検知が永久に起きない**
- 最悪、誰かが誤ったアドレス宛に送金すると**トークンが永久に消失**する

コントラクトアドレスの1文字ミスは、バグではなく**資金喪失**に直結する。

---

## マルチエージェントレビューが機能した理由

ブロックチェーン専門エージェントが Etherscan で実際のアドレスを照合し、コードと一致しないことを検出した。

人間のコードレビューで `BF9` と `BDB` の違いを見つけるのは非常に困難。専門エージェントが「このアドレスを実際に検索してみる」というアプローチを取ったからこそ検出できた。

---

## もう一つの気づき：Infura URL のログ漏洩リスク

Web3jConfig で `new HttpService(clientAddress)` と書くだけだと、OkHttp が起動時に接続先 URL をログに出力するケースがある。

URL の中に Infura の API Key が含まれているため、**ログファイルを見られると API Key が漏れる**。

対策として `HttpLoggingInterceptor.Level.NONE` を設定した。

```java
OkHttpClient httpClient = new OkHttpClient.Builder()
    .addInterceptor(new HttpLoggingInterceptor().setLevel(HttpLoggingInterceptor.Level.NONE))
    .build();
```

これは「コードに API Key を書かない」だけでは防げない類のリスク。**ログへの間接的な漏洩**を常に意識する必要がある。

---

## 得た教訓

| 教訓 | 詳細 |
|---|---|
| コントラクトアドレスは必ず外部ソース（Etherscan）と照合する | 目視確認は信頼できない |
| 秘密情報はコードだけでなくログにも漏れうる | URL・ヘッダー・例外メッセージを疑う |
| 専門エージェントを複数立てる意味がある | 一つの視点では見逃すものがある |
| レビューは PR を出す前にも価値がある | 早く指摘されるほど修正コストが低い |

---

## 今後に活かすこと

- コントラクトアドレスを変更・追加するときは必ずブロックチェーンエージェントに Etherscan 照合を依頼する
- `scripts/security-check.sh` を PR 前に必ず実行する
- 外部サービスとの接続設定（URL・認証情報）はログ出力を明示的に `NONE` にする
