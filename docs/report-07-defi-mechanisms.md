# レポート07 — DeFi の仕組み：AMM・Uniswap・流動性プール

## はじめに

Phase 5 で実装した EIP-2612 Permit は「ガスレスでトークンを移動させる」技術だった。  
DeFi（分散型金融）はその延長線上にある世界で、Permit のような「承認の委譲」を使って  
**銀行なしで両替・貸し借り・利回り運用**を実現している。

本レポートでは「DeFi がどう動くか」を AMM とUniswap を中心に整理する。

---

## 1. DeFi とは何か

### 従来の金融（CeFi）との比較

```
従来の金融（CeFi: Centralized Finance）:
  ユーザー → 銀行 → 両替所 → 決済ネットワーク → 相手
  ↑ 各ステップに中間業者がいて、手数料・審査・営業時間の制約がある

DeFi（Decentralized Finance）:
  ユーザー → スマートコントラクト → 相手
  ↑ 中間業者なし。コードが規則。24時間365日稼働
```

DeFi の主なサービス分類：

| カテゴリ | 代表プロトコル | できること |
|---|---|---|
| DEX（分散型取引所） | Uniswap, Curve | トークンを別のトークンに両替 |
| 貸し借り | Aave, Compound | トークンを担保に借り入れ |
| ステーブルコイン | MakerDAO, Frax | 暗号資産担保で$1トークンを発行 |
| リキッドステーキング | Lido | ETH をステーキングしながら流動性を保持 |
| Yield Aggregator | Yearn | 最高利回りを自動で選択 |

## 2. DEX とオーダーブックの違い

### 従来の取引所（オーダーブック方式）

```
買い注文 (Bid):           売り注文 (Ask):
  $2490 × 100 USDC           $2500 × 50 USDC
  $2480 × 200 USDC           $2510 × 150 USDC
  $2470 × 500 USDC           $2520 × 300 USDC

マッチングエンジン（中央集権サーバー）が買いと売りを突き合わせる
```

問題点：
- マーケットメーカー（流動性提供者）が必要
- 注文が薄い通貨ペアは取引できない
- 中央サーバーがダウンすると取引できない

### AMM（Automated Market Maker / 自動マーケットメーカー）

オーダーブックを使わず、**数学的な式**で常に価格を決定する仕組み。

```
プール: USDC 1,000,000 ← → ETH 400

価格 = USDC量 ÷ ETH量 = 1,000,000 ÷ 400 = 2,500 USDC/ETH

注文の突き合わせは不要。式が価格を自動計算する。
```

## 3. x × y = k の数学

Uniswap V2 が使う公式：

```
x × y = k（定数積公式）

x = トークンAの量
y = トークンBの量
k = 定数（流動性の大きさを表す）
```

### 具体例：ETH/USDC プール

初期状態：
```
ETH: 400  ×  USDC: 1,000,000  =  k = 400,000,000
価格: 1 ETH = 2,500 USDC
```

ユーザーが 1 ETH を売る（ETH を入れて USDC を出す）：

```
ETH が 400 → 401 に増える
k = 400,000,000 は変わらない

新しい USDC = 400,000,000 ÷ 401 = 997,506 USDC

受け取り = 1,000,000 - 997,506 = 2,494 USDC

価格は 2,500 → 2,494 に下がった（スリッページ）
```

### スリッページが起きる理由

```
小さいプール（流動性が低い）:
  ETH: 4  ×  USDC: 10,000  =  k = 40,000

  1 ETH を売る → 新ETH = 5
  新USDC = 40,000 ÷ 5 = 8,000 USDC
  受け取り = 10,000 - 8,000 = 2,000 USDC（本来2,500のはずが20%も安い）

大きいプール（流動性が高い）:
  ETH: 40,000  ×  USDC: 100,000,000  =  k = 4,000,000,000,000

  1 ETH を売る → 新ETH = 40,001
  新USDC ≈ 99,997,500 USDC
  受け取り ≈ 2,500 USDC（ほぼ理論値）
```

流動性が高いほどスリッページが小さくなる → **流動性提供が重要な理由**。

## 4. 流動性プール（Liquidity Pool）と LP トークン

### 流動性提供者（LP: Liquidity Provider）

誰でもプールにトークンを預けることができる。

```
流動性提供:
  ユーザーが ETH 1 + USDC 2,500 をプールに預ける
    ↓
  LP トークン（プールへの持ち分証明）を受け取る
    ↓
  取引手数料（0.3%）をプール持ち分に応じて受け取る
```

LP トークンは ERC-20 トークンであり、それ自体を別のプロトコルで使える（DeFi の「コンポーザビリティ」）。

### 手数料の仕組み

```
プール: ETH 400 + USDC 1,000,000（手数料 0.3%）

ユーザーが 1 ETH を売る:
  0.997 ETH 分が k 計算に入る（0.003 ETH = 0.3% は手数料として残る）

手数料はプールに蓄積 → LP の持ち分価値が増える
```

## 5. インパーマネントロス（一時的損失）

LP（流動性提供者）が直面する特有のリスク。

### 具体例

```
初期: ETH = 2,500 USDC で流動性提供
  預けた資産: ETH 1 + USDC 2,500（合計 5,000 USDC 相当）

1週間後: ETH = 3,600 USDC に上昇

プール内部の比率変化（k = 一定）:
  ETH: 1 × USDC: 2,500 → ETH: 0.833 × USDC: 3,000
  （アービトラージャーが ETH を買ってプールを均衡させた結果）

プールから引き出した場合:
  ETH 0.833 × 3,600 + USDC 3,000 = 5,999 USDC

単純保持していた場合:
  ETH 1 × 3,600 + USDC 2,500 = 6,100 USDC

インパーマネントロス = 6,100 - 5,999 = 101 USDC（約1.65%の機会損失）
```

「インパーマネント（一時的）」なのは、価格が元に戻ると損失が消えるから。  
しかし実際には元に戻らないことも多く、手数料収入と比較してどちらが得かを判断する必要がある。

## 6. Uniswap のバージョン進化

### Uniswap V2（2020年）— 基本形

```
x × y = k の単純な定数積式
全価格帯（0〜∞）に均等に流動性を分散
```

問題: 資本効率が悪い。1 ETH = 2,500 USDC なのに、0.01 USDC や 100,000 USDC  
の価格帯にも流動性が置かれている（実際には使われない）。

### Uniswap V3（2021年）— 集中流動性

```
LP が「この価格帯だけに流動性を置く」を指定できる

例: 「ETH が 2,400〜2,600 USDC の間だけ流動性を提供する」

メリット: 指定範囲内では資本効率が最大100倍向上
デメリット: 価格が範囲外に出ると手数料を稼げない（リバランスが必要）
```

### Uniswap V4（2024年）— フック機構

```
「取引前後にカスタムロジックを実行できる」Hooks 機能を追加

例えば:
- 取引前に KYC チェック
- 特定の時間帯だけ手数料を変更
- 価格が一定以上変動したら手数料を引き上げ（MEV 対策）
```

## 7. MEV（最大抽出可能価値）

### ブロック生成者の有利な立場

```
通常ユーザー: トランザクションを mempool（未確認TX プール）に送信
バリデータ: どの TX をどの順番でブロックに入れるかを決定できる
```

悪意あるバリデータ（またはボット）は：

```
① ユーザーの「ETH を売る」TX を mempool で発見
② 自分の TX を先に入れて ETH を安く買う（フロントランニング）
③ ユーザーの TX が実行されて ETH 価格が上がる
④ 直後に高く売る（バックランニング）

→ サンドイッチ攻撃: ユーザーが損をして MEV ボットが儲ける
```

対策：スリッページ耐性設定（許容範囲外なら TX を失敗させる）、  
Flashbots などのプライベート TX ルーティング。

## 8. このプロジェクトとの関係

### EIP-2612 Permit → DeFi の組み合わせ

このプロジェクトで実装した Permit は、DeFi との連携の入口になる。

```
現在の実装:
  ① JPYC 保有者が Permit 署名
  ② スペンダー（サーバー）が permit() + transferFrom() を実行
  → 決済が完了

DeFi 連携の拡張イメージ:
  ① JPYC 保有者が Permit 署名
  ② Uniswap Router コントラクトに Permit を渡す
  ③ Router が JPYC → USDC に自動スワップ
  ④ USDC で商品代金を支払い

メリット: 「JPYC 建ての請求書を USDC で支払える」
→ ユーザーの保有トークンに関わらず決済できる
```

### JPYC の DeFi エコシステム

Polygon では JPYC を使った DeFi が存在する：

```
QuickSwap（Uniswap fork on Polygon）:
  JPYC/USDC プール → 円建て資産をドル建て資産と交換できる

Aave（貸し借りプロトコル）:
  JPYC を担保に MATIC を借りる → ガス代を確保しながら JPYC を保有し続ける
```

## 9. なぜガス代に注目するのか

DeFi 取引ではコントラクト呼び出しが複数連鎖するため、ガス代が重要。

```
Uniswap V3 でのスワップ一回のガス使用量:
  Ethereum Mainnet: 約 120,000〜200,000 gas
  → baseFee 10 gwei 時: 0.002 ETH ≈ 600円

Polygon:
  Polygon PoS: 約 120,000 gas
  → baseFee 30 gwei 時: 0.0036 POL ≈ 0.1円（1/6000）
```

Polygon が DeFi に適している理由：同じロジックでもガス代が Ethereum の 1/1000 以下。

## まとめ

```
DEX の仕組み:
  オーダーブック不要 → AMM（x × y = k）が自動で価格を決定
  流動性プールに誰でも資金を預け → 手数料収入を得る

インパーマネントロス:
  価格変動が大きいほど LP の機会損失が発生
  手数料収入 vs 損失 のトレードオフ

MEV:
  ブロック生成者がTX順序を操作して利益を得る問題
  スリッページ設定・プライベートTXで軽減

EIP-2612 Permit との接点:
  Permit 署名 → DEX ルーターに渡す → ガスレスでスワップ + 決済
  → 「どのトークンを持っていても決済できる」UX の実現
```

DeFi は「コードがルール」の世界。中間業者がないからこそ、  
コントラクトの数学的な正確さとセキュリティが決定的に重要になる。
