# レポート11 — なぜ Ethereum を選ぶのか：全チェーン徹底比較

> 作成日: 2026-05-31
> テーマ: 数あるブロックチェーンの中からなぜ Ethereum（および Ethereum エコシステム）を選ぶのか、技術・エコシステム・分散化の3軸で深掘りする

---

## はじめに — 「なぜ Ethereum か」という問いの本質

ブロックチェーンは今や 500 以上存在する。
「Bitcoin があるのになぜ Ethereum？」「Solana の方が速くて安いのでは？」という疑問は正当だ。

本レポートでは、実際にこのプロジェクトで JPYC（Polygon）を使う判断に至った背景を説明しながら、各チェーンの本質的な違いを整理する。

---

## 1. ブロックチェーンの「世代」を理解する

まず大きな地図を描く。ブロックチェーンは世代によって「できること」が根本的に異なる。

```
第1世代: Bitcoin（2009）
  └─ 目的：デジタル通貨の移転のみ
  └─ スクリプト：限定的なプログラム（ループ不可・チューリング不完全）
  └─ できないこと：複雑なロジック、条件分岐のある契約

第2世代: Ethereum（2015）
  └─ 目的：汎用スマートコントラクトプラットフォーム
  └─ EVM（Ethereum Virtual Machine）：チューリング完全な仮想マシン
  └─ できること：条件付き送金・投票・NFT・DeFi・何でも

第3世代: Solana / Avalanche / BNB Chain など（2020年代）
  └─ 目的：速度・コストの改善
  └─ 多くは Ethereum の設計思想を引き継ぎつつ性能向上
```

**結論から言うと**：速度や手数料だけで選ぶと本質を見誤る。
「何のためのブロックチェーンか」を先に考えなければならない。

---

## 2. Bitcoin との比較 — 「通貨」と「プラットフォーム」の違い

### Bitcoin は「デジタルゴールド」

Bitcoin の設計思想は明確だ：**価値の保存と移転**に特化する。

```
Bitcoin のスクリプト（限定的）:
  - 「AからBへ1BTC送る」ならできる
  - 「1000円以上の注文が確認されたら送金する」はできない
  - ループ処理・複雑な条件分岐は意図的に排除されている
```

これは「バグを排除するための設計判断」であって欠点ではない。
だが**決済バックエンド**として使うには根本的に力不足だ。

| 機能 | Bitcoin | Ethereum |
|------|---------|----------|
| 単純な送金 | ✅ | ✅ |
| ERC-20 トークン発行 | ❌ | ✅ |
| 条件付き送金（Permit） | ❌ | ✅ |
| オンチェーンで注文管理 | ❌ | ✅ |
| 入金イベントの検出（Transfer Event） | ❌ | ✅ |
| ステーブルコインの発行・流通 | ❌ | ✅ |

JPYC が Ethereum エコシステム上に存在する理由もここにある。
ERC-20 規格があってはじめて「1円ペッグのトークン」を発行・流通させられる。

---

## 3. Solana との比較 — 「速さ」の代償

Solana は現時点でおそらく最速のブロックチェーンだ。

```
処理速度の比較:
  Bitcoin:  ~7 TPS（1秒あたりのトランザクション数）
  Ethereum: ~15〜30 TPS（L1のみ）
  Solana:   ~50,000〜65,000 TPS（理論値）
  Visa:     ~24,000 TPS（参考）
```

これだけ見ると「Solana を使えばいい」に見える。だが実態は：

### Solana の問題点 ①：停止歴が多い

```
主なダウン事例:
  2021年9月:  17時間停止
  2022年1月:  30時間停止（バリデータのメモリ不足）
  2022年5月:  7時間停止
  2023年2月:  19時間停止
  ……（複数回）
```

Ethereum のメインネットが「長時間停止した」事例はほぼない。
決済システムとして「止まらないこと」は最重要要件であり、ここは決定的な差だ。

### Solana の問題点 ②：ハードウェアの集中化

Solana が速い理由の一つは「バリデータに高性能マシンが必要なこと」だ。

```
Solana バリデータ推奨スペック:
  CPU: 24コア以上
  RAM: 256GB以上
  SSD: RAID対応の高速NVMe
  帯域: 300Mbps以上
  コスト: 月数十万円規模
```

これでは個人がバリデータを運用できない。
バリデータが集中すると「少数者による合意」→ 検閲・停止リスクが上がる。

### Solana の問題点 ③：エコシステムの非互換

EVM（Ethereum Virtual Machine）のコードは Solana では動かない。

```
Ethereum のスマートコントラクト（Solidity）
  → Polygon で動く ✅（EVM互換）
  → Arbitrum で動く ✅（EVM互換）
  → Optimism で動く ✅（EVM互換）
  → Solana では動かない ❌（独自のプログラムモデル）
```

JPYC を Polygon で使えるのは Polygon が EVM 互換だからだ。
Solana エコシステムへの移植には根本的な書き直しが必要になる。

---

## 4. BNB Chain（Binance Smart Chain）との比較 — 「中央集権」の問題

BNB Chain は Binance（世界最大の暗号資産取引所）が運営するチェーンだ。

```
メリット:
  - 手数料が安い（$0.01〜0.10程度）
  - Ethereum とほぼ同じ EVM 互換
  - Binance エコシステムとの連携

デメリット（決定的な問題）:
  - バリデータは 21 ノードのみ（Ethereum は約1万ノード）
  - Binance が 21 ノードのうち多数を実質的にコントロール
  - 「分散型台帳」を名乗るが事実上 Binance が管理者
```

2022年10月、BNB Chain は攻撃を受けて約7億ドルが盗まれかけた。
対応として**バリデータに投票でチェーンを停止させた**。

これはある意味「素早い対応」だが、別の意味では「中央集権システムと同じ」だ。
Ethereum は誰にも止められない。それが設計の中心にある。

### 決済システムとしての信頼性

```
中央集権型決済（Visa・銀行）:
  運営者 → 止められる・規制できる・検閲できる

BNB Chain:
  Binance   → 実質的に止められる・規制できる

Ethereum:
  誰も      → 止められない・検閲できない（Immutability）
```

「誰も止められない」ことを「強み」と捉えるかどうかはユースケース次第だが、**金融インフラとして中立性**を求めるなら Ethereum の分散化は大きな価値がある。

---

## 5. Avalanche・Cosmos・Polkadot との比較

### Avalanche

```
強み:
  - サブネット設計（アプリ専用のチェーンを作れる）
  - 最終確定が非常に速い（<2秒）
  - EVM互換のC-Chainが存在する

弱み:
  - 開発者・プロジェクト数がEthereumの1/10以下
  - C-Chainのエコシステムはまだ小さい
  - ステーブルコインの流動性が薄い
```

### Cosmos / IBC

```
強み:
  - チェーン間の相互運用（IBC プロトコル）
  - アプリチェーン（独自チェーンを立ちやすい）

弱み:
  - 「Cosmos エコシステム」内でしか繋がらない
  - Ethereum の ERC-20 エコシステムとは直接相互運用できない
  - 開発者の絶対数が少ない
```

これらは技術的に面白いが、**ステーブルコイン決済の文脈**では Ethereum エコシステムのスタンダードから外れると「決済相手との合意コスト」が激増する。

## 6. Ethereum の L2 と Polygon — 「拡張」という発想

ここが本プロジェクトで JPYC を扱う上で最も重要な部分だ。

### なぜ JPYC は Polygon にあるのか

JPYC は元々 Ethereum メインネット上に存在した（v1）。
しかし Ethereum L1 のガス代問題から Polygon（v2）へ移行した。

```
Ethereum L1 のガス代:
  ERC-20 Transfer: $5〜50（混雑時）
  Permit + Transfer: $10〜100
  → 「100円の決済に50円の手数料」は非現実的

Polygon のガス代:
  ERC-20 Transfer: $0.001〜0.01
  Permit + Transfer: $0.005〜0.05
  → 実用的な決済が可能
```

### Polygon とは何か

Polygon は Ethereum の「サイドチェーン / L2」だ。

```
Ethereum L1 (セキュリティ基盤)
    ↑ブリッジ
Polygon (高速・低コストの実行レイヤー)
    └── JPYC など多くのトークンが存在
    └── EVM 完全互換（Solidity コードをそのまま動かせる）
    └── Ethereum のセキュリティモデルを一部継承
```

Polygon で書いた Solidity コードは Ethereum L1 でも動く。
これが「Ethereum エコシステムを選ぶ」意味だ。Polygon を選ぶことは Ethereum を選ぶことと同義だ。

### Ethereum の L2 ロールアップ一覧

```
Optimistic Rollup:
  Optimism   → OP Mainnet、Coinbaseの Base など
  Arbitrum   → DeFiの流動性が特に厚い

ZK Rollup（Zero-Knowledge Rollup）:
  zkSync Era
  Starknet
  Polygon zkEVM（Polygon自身もL2へ移行中）

全て EVM 互換 → Ethereum のコード資産がそのまま使える
```

## 7. 開発者エコシステムの規模

技術選定で見落とされがちな「エコシステムの厚み」を数字で見る。

```
GitHubのSolidityリポジトリ数（2024年時点）:
  Ethereum/EVM: 約 50万以上のリポジトリ
  Solana: 約 3〜5万
  BNB Chain: Ethereumのコピーが多数

Stack OverflowのQ&A数:
  Ethereum/Solidity: 約 10万件以上
  Solana/Rust: 数千件

主要ライブラリの成熟度:
  Web3j（Java）   → Ethereum専用、10年以上の歴史
  ethers.js       → Ethereum/EVM標準ライブラリ
  viem/wagmi      → React向けEthereumライブラリ（本プロジェクトのフロント用）
```

「ライブラリが豊富 → 問題解決が速い → 開発コストが低い」
この連鎖は実際のプロジェクトで非常に大きい。

## 8. 標準規格の集積地

なぜ Ethereum に USDC・USDT・DAI・JPYC が集まるのか。
それは「EIP（Ethereum Improvement Proposal）」という規格策定プロセスによる。

```
ERC-20  (EIP-20):  トークン標準（すべてのステーブルコインの基礎）
ERC-721 (EIP-721): NFT 標準
ERC-1155 (EIP-1155): マルチトークン標準
EIP-2612 (Permit): ガスレス承認（本プロジェクトの核心機能）
EIP-3009:          transferWithAuthorization（Circle の USDC が採用）
EIP-712:           型付き署名標準（Permit・SIWEの基礎）
EIP-4337:          アカウント抽象化（ウォレットの次世代設計）
```

これらの規格が**業界標準**として確立されているのは Ethereum だけだ。
他のチェーンの「ERC-20 互換」は「Ethereum の ERC-20 規格をコピーした」という意味であり、Ethereum がオリジネーターだ。

---

## 9. 分散化の定量的評価

「分散化されている」を定量的に評価する指標として Nakamoto 係数がある。

```
Nakamoto 係数 = ネットワーク制御に必要な最小バリデータ数

  Bitcoin:    ~4〜7（マイニングプール上位数社で過半数）
  Ethereum:   ~3〜5（ただしステーキングプールが問題）
  Solana:     ~19（バリデータの上位19で 33%超）
  BNB Chain:  ~7（21バリデータのうち実質Binance管理）
  Polygon:    ~10〜20

※ 低いほど集中化している（Nakamoto 係数は高い方が分散）
```

注意：Ethereum も Lido（ステーキングプール）の集中化問題を抱えている。
「完全に分散化されたチェーン」は存在しない。
しかし 1万以上のノードが稼働する Ethereum は他チェーンと比べて断然多い。

## 10. 結論 — なぜ Ethereum（エコシステム）を選ぶのか

| 評価軸 | Ethereum | Bitcoin | Solana | BNB Chain |
|--------|----------|---------|--------|-----------|
| スマートコントラクト | ◎ | △ | ○ | ○ |
| 分散化・検閲耐性 | ◎ | ◎ | △ | △ |
| 停止耐性 | ◎ | ◎ | △ | ○ |
| エコシステムの厚み | ◎ | ○ | △ | △ |
| 標準規格の充実 | ◎ | △ | △ | △（コピー） |
| ステーブルコインの流動性 | ◎ | ✗ | ○ | ○ |
| ガスコスト（L1） | △ | △ | ◎ | ◎ |
| L2 による拡張性 | ◎ | △ | ✗ | △ |
| 日本円決済（JPYC） | ◎（Polygon）| ✗ | ✗ | ✗ |

**Ethereum を選ぶ5つの理由：**

1. **スマートコントラクトの元祖** — ERC-20・EIP-2612(Permit)・EIP-712 すべてここが起点
2. **最大のエコシステム** — ライブラリ・事例・開発者が他の追随を許さない
3. **EVM 互換 L2 の拡張性** — コストはL2(Polygon等)で解決、資産はEthereumエコシステムにある
4. **JPYC の存在場所** — 日本円ステーブルコインは Polygon（EVM互換）上に存在する
5. **検閲耐性・不変性** — 誰も止められない金融インフラとしての中立性

ガスコストの問題は「Polygon・L2 を使う」で解決できる。
エコシステムの薄さは「他のチェーンに移ることで」起こる問題で、解決が難しい。

だから **Ethereum エコシステムを中心に選ぶことは、長期的に最もリスクが低い選択** だ。

## 参考：「Ethereum キラー」の現在地

2020〜2022年に「Ethereum を倒す」と言われたチェーンの現在地を振り返ると：

```
EOS:       一時期1兆円の時価総額 → 現在ほぼ忘れられた
TRON:      USDT の大量流通で存在感 → 中央集権的・Ethereum に到達できず
Cardano:   厳密な学術設計 → 実用DeFiエコシステムの形成に遅れ
NEO:       中国版Ethereum と呼ばれた → 急速に縮小
```

「技術的に正しい」ことと「採用される」ことは別問題だ。
Ethereum の最大の強みは「技術」だけでなく「10年以上の実績と信頼」にある。
