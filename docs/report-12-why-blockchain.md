# レポート12 — なぜブロックチェーンが必要か：既存システムとの本質的な違い

> 作成日: 2026-05-31
> テーマ: 「銀行振込やクレジットカードがあるのになぜブロックチェーンか」という問いに正面から答える

---

## はじめに — 「そもそも何の問題を解くのか」

新しい技術を評価するとき、最初に問うべきは「それは何の問題を解くのか」だ。

ブロックチェーンが解こうとしている問題は一言で言うと：

> **「信頼できる第三者なしに、価値を移転・記録・プログラムできるか」**

この問いを理解するために、まず「なぜ第三者への信頼が必要になるのか」から始めよう。

---

## 1. 現在の決済システムの仕組み

### 銀行振込が「届く」仕組み

AさんがBさんに1万円を振り込むとき、実際に起きていることは：

```
Aさん（三菱UFJ口座）→ Bさん（三井住友口座）

実際の処理:
  1. Aさんの三菱UFJ口座の残高を -10,000円
  2. 三菱UFJ ↔ 全国銀行データ通信システム（全銀ネット）
  3. 全銀ネット ↔ 三井住友
  4. Bさんの三井住友口座の残高を +10,000円

「お金」が動いているのではなく、
「各銀行のデータベース上の数字」が変わっているだけ
```

これは本質的に「中央のデータベースに対する書き込み権限を持つ機関が、数字を書き換えている」に過ぎない。

### 私たちは何を信じているのか

```
銀行口座の残高 = 銀行のデータベースに書かれた数字
クレジット残高 = カード会社のデータベースに書かれた数字
証券口座の株 = 証券保管振替機構のデータベースに書かれた数字
```

私たちは「その数字を正確に管理してくれる機関」を信じている。
これを**信頼ある第三者（Trusted Third Party）**と呼ぶ。

信頼できる第三者がいるならば、現行システムは非常によく機能する。
問題は：**その信頼が崩れたとき、または第三者が存在しないとき**だ。

## 2. 既存システムが抱える5つの本質的な問題

### ① 許可が必要（Permission-based）

```
銀行口座を開くには:
  - 身分証明書が必要
  - 国籍・居住地の制約がある
  - 銀行に「許可」してもらわなければならない

世界の非銀行口座人口:
  約 14億人（世界銀行 2021年データ）
  → スマートフォンは持っているが銀行口座は持てない人々
```

Ethereum のウォレットは「許可なし」で作れる。
秘密鍵を生成した瞬間、誰でも世界中のブロックチェーンと接続できる。

### ② 速度と時間の制約

```
国際送金（SWIFT）:
  所要時間: 2〜5営業日
  手数料: $25〜50（+為替コスト）
  条件: 銀行の営業時間内

Ethereum の送金:
  所要時間: 12秒〜数分（ファイナリティまで）
  手数料: ガス代のみ（L2で $0.01以下も可能）
  条件: 24時間365日、休みなし
```

日本の全銀システムでさえ、2023年まで夜間・休日は送金が翌日扱いだった（全銀モアタイム稼働で改善）。

### ③ 検閲可能性（Censorship）

```
中央集権型システムの検閲例:
  - 政府命令による口座凍結
  - 取引の一方的な停止・差し戻し
  - 特定の個人・組織への送金ブロック
  - 資産の没収

実際の事例:
  2022年: カナダ政府がトラック運転手デモ参加者の口座を凍結
  2022年: SWIFT がロシアを排除（国際送金ほぼ不可に）
  2018年: Visa・Mastercard がウィキリークスへの寄付をブロック
```

Ethereum 上のトランザクションは誰も止められない。
（ただし「止められない」ことが規制当局との摩擦を生む点は後述）

### ④ 透明性の欠如

```
銀行のシステム:
  「あなたのお金は正しく管理されています」← 信じるしかない
  帳簿の中身を一般公開していない
  外部から検証する手段がない

リーマンショック（2008年）の本質:
  複雑な金融商品の中身を誰も把握していなかった
  → 信頼が崩れた瞬間、世界金融危機に
```

Ethereum のすべてのトランザクションはブロックエクスプローラーで誰でも確認できる。
JPYC の送金記録も、スマートコントラクトのコードも、すべて公開されている。

### ⑤ プログラマビリティの欠如

```
銀行振込でできること:
  「AからBに○円送る」だけ

スマートコントラクトでできること:
  「条件Xが成立したときだけAからBに○円送る」
  「30日ごとに自動で○円送る」
  「担保がXを下回ったら清算する」
  「NFTの所有者が変わったら前所有者に3%ロイヤリティを払う」
```

これを「プログラマブルマネー」と呼ぶ。
本プロジェクトの **Permit機能**（事前署名で後でガスを払わずに承認）も、スマートコントラクトのプログラマビリティがあってこそ実現できる。

## 3. 「二重支払い問題」とブロックチェーンの解決策

デジタルデータの最大の問題：**コピーが無限にできること**。

```
物理的なお金:
  1万円札を渡したら、自分の手元からなくなる（物理法則）

デジタルデータ:
  ファイルを送っても、自分の手元にもコピーが残る
  → 同じデジタルお金を2人に送ることが「技術的に」可能
  → これを「二重支払い（Double Spend）」という
```

従来の解決策は「中央のデータベースが正の管理者」になること。
銀行のシステムが「あなたの残高は1万円で、すでに送金済みなら引き落とし済み」と管理する。

### ブロックチェーンによる解決

中央のデータベースなしに二重支払いを防ぐ方法：

```
ビットコインの解決策（Proof of Work）:
  1. 全参加者が取引の履歴を持つ（分散台帳）
  2. 新しいブロックに取引を追加する権利を
     「複雑な計算問題を解くこと」で競わせる（マイニング）
  3. 最も長いチェーンが「正しい歴史」とみなされる
  4. 二重支払いをするには過去のブロックをすべて書き換えが必要
     → 全ハッシュレートの51%を支配しなければならない（現実的不可能）

Ethereum の解決策（Proof of Stake）:
  1. バリデータは ETH を担保（ステーク）として預ける
  2. 不正をしたら担保（スラッシング）
  3. 経済的インセンティブで正直な行動を強制
  4. 二重支払いの試みは担保没収リスクを伴う
```

これが「信頼ある第三者なしに価値を移転できる」理由だ。
**「数学と経済的インセンティブ」が第三者の代わりを務める。**

## 4. スマートコントラクトが解決する「契約の自動執行問題」

### 従来の契約の問題

```
家賃の自動引き落とし：
  実現できているが、銀行システムの「許可」と「手数料」が必要
  海外送金では事実上不可能

キックスターター型の資金調達：
  「目標金額に達しなかったら全額返金」
  → エスクロー業者が必要（手数料・信頼問題）

著作権ロイヤリティ：
  CDが売れたら作曲家に2%払う
  → 中間業者なし・自動化は困難（現実には多くの未払いが発生）
```

### スマートコントラクトによる自動執行

```solidity
// 「目標金額に達したら受取人へ、未達なら返金」の概念コード
contract Crowdfunding {
    mapping(address => uint) public contributions;
    uint public goal;
    uint public deadline;
    address public creator;

    function contribute() public payable {
        contributions[msg.sender] += msg.value;
    }

    function withdraw() public {
        if (address(this).balance >= goal) {
            // 目標達成: クリエイターに送金
            payable(creator).transfer(address(this).balance);
        } else if (block.timestamp > deadline) {
            // 未達: 返金
            payable(msg.sender).transfer(contributions[msg.sender]);
        }
    }
}
```

このコードはデプロイされたら**誰も止められない**。
クリエイターが気が変わっても、国が禁止しても、コードは動き続ける。

本プロジェクトの文脈で言えば：
- **Transfer イベントの自動検知** → 入金を自動で確認
- **Permit による事前承認** → ユーザーの署名だけで後から引き落とし可能
- これらはすべて「仲介者なしに、コードが契約を執行する」構造だ

## 5. 決済バックエンドにブロックチェーンを使う具体的な価値

本プロジェクト（JPYC 決済バックエンド）の文脈で整理する。

### 価値 ① 不変の取引証跡

```
オンチェーンの記録:
  txHash: 0x7a6b1c...（これが世界中のノードに記録される）
  From: 0xABC...（送金者）
  To:   0xDEF...（受取人）
  Value: 1000 JPYC
  Block: 52,341,234
  Timestamp: 2026-05-31 11:39:56 UTC

この記録は:
  - 誰にも削除できない
  - 誰にも改ざんできない
  - 世界中のノードが同一の記録を持つ
  - ブロックエクスプローラーで誰でも検証できる
```

### 価値 ② 24/7/365 の自動化

```
従来の EC サイトの入金確認フロー:
  1. お客様が振込
  2. 銀行の翌営業日処理
  3. 担当者が入金確認（営業時間内）
  4. 注文処理開始
  所要時間: 1〜2営業日

JPYC 決済バックエンドのフロー:
  1. お客様が Transfer 実行（またはPermitで署名）
  2. Transfer Event をポーリングで検知（数分以内）
  3. PaymentOrder を CONFIRMED に自動更新
  4. 注文処理開始
  所要時間: 数分（ブロックの確認数次第）
```

### 価値 ③ プログラマブルな決済条件

```
Permit の仕組み（本プロジェクトの核心）:

  ユーザーが署名するだけ（ETH 不要・ガス代負担なし）
  ↓
  バックエンドが Permit を呼び出し（バックエンドがガスを支払う）
  ↓
  transferFrom で JPYC をユーザーからマーチャントへ
  ↓
  ユーザーは ETH を一切持っていなくても JPYC で決済できる

これは「クレジットカードのオーソリ（仮売上）→ キャプチャ（実売上）」に相当するが、
中央のカード会社なしに実現されている。
```

### 価値 ④ グローバル・ボーダーレス

```
従来の国際決済（SWIFT）:
  手数料: $25〜50
  時間: 2〜5営業日
  制約: AML/KYC・制裁リスト確認
  書類: 多数

JPYC on Polygon:
  手数料: $0.01 以下
  時間: 数秒〜数分
  制約: ウォレットアドレスのみ（KYCはオフチェーンで実施可能）
  書類: 不要
```

## 6. ブロックチェーンが「最適解ではない」ケース

誠実に欠点も整理する。ブロックチェーンが向かない場合：

```
向かないケース:
  ① 高スループットが必要（毎秒10万件以上）→ L1 Ethereum では無理
  ② 高頻度の状態変更 → ガス代が膨大になる
  ③ 個人情報の保存 → ブロックチェーンはパブリック（プライバシー問題）
  ④ 完全なプライバシーが必要 → ZK証明などの追加レイヤーが必要
  ⑤ 規制が許可制を求める → 非許可型ブロックチェーンとの齟齬

「従来型DB で十分なケース」:
  - 単一企業内のデータ管理
  - 信頼できる相手同士のシステム
  - 高頻度・小額の大量決済
```

ブロックチェーンは「万能薬」ではなく、「特定の問題に対する特定の解」だ。
**中央集権的な管理者への信頼が問題になる文脈**で最大の価値を発揮する。

## 7. Web3 とは何か

「ブロックチェーンが必要か」の議論を踏まえて、Web3 の定義を整理する。

```
Web1.0（1990年代〜）:
  読むだけのウェブ
  → 静的なHTMLページ、一方向の情報発信

Web2.0（2000年代〜）:
  読む + 書く
  → SNS・YouTube・Wikipedia・ユーザー生成コンテンツ
  問題: データは GAFA のサーバーに集中
         → 検閲・プラットフォームリスク・プライバシー侵害

Web3（2020年代〜）:
  読む + 書く + 所有する
  → ブロックチェーン上でデータ・資産・IDを「自分で所有」
  実現手段: スマートコントラクト・NFT・DeFi・ウォレット
```

### Web3 の核心概念

| 概念 | 説明 | 本プロジェクトでの対応 |
|------|------|----------------------|
| **ウォレット** | 秘密鍵を管理し、署名・送金を行うソフトウェア | MetaMask・HashPort Wallet |
| **DeFi** | 銀行なしの金融サービス（貸借・取引所・保険） | 参考知識（report-07） |
| **NFT** | 非代替トークン（所有権の証明） | 参考知識 |
| **DAO** | 分散型自律組織（投票でコード変更） | 参考知識 |
| **SIWE** | ウォレット署名によるログイン | `SiweService.java` |
| **Permit** | EIP-2612 ガスレス承認 | `PermitService.java` |

Web3 はひとつの「思想」であり、「ユーザーが自分のデータ・資産・IDを支配する」という方向性を持つ。
ブロックチェーンはその技術的な基盤だ。

## 8. 結論 — ブロックチェーンが「必要」な状況

```
ブロックチェーンが有効な状況（チェックリスト）:

  ✅ 「信頼できる第三者」が存在しない、またはコストが高い
  ✅ 取引の透明性・監査可能性が重要
  ✅ 自動執行（スマートコントラクト）で中間業者を削除できる
  ✅ グローバルなアクセスが必要（銀行口座不要）
  ✅ 24/7/365 の稼働が必要
  ✅ 検閲耐性・不変性が価値を持つ

本プロジェクトへの適用:
  ✅ JPYC による「プログラマブルな日本円決済」
  ✅ Permit による「中間業者なしのガスレス承認」
  ✅ Transfer Event による「改ざん不能な入金証跡」
  ✅ SIWE による「パスワードなしのウォレット認証」
```

ブロックチェーンは「銀行を不要にする技術」ではなく、
「特定の文脈で、信頼のコスト・速度・プログラマビリティを根本的に変える技術」だ。

その文脈の一つが「ステーブルコインによる決済バックエンド」であり、それがこのプロジェクトの存在理由だ。
