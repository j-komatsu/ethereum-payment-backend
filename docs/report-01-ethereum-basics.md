## 1. ブロックチェーンとは何か

### なぜ「チェーン」という名前か

ブロックチェーンは「**ブロック**（データのまとまり）」が「**チェーン**（鎖）」のようにつながったデータ構造です。

```
[Genesis Block] → [Block #1] → [Block #2] → [Block #3] → ...
```

各ブロックは「直前のブロックのハッシュ値」を内部に持っています。
これによって「ブロックが鎖のようにつながっている」状態になります。

### ブロックの中身には何が入っているか

```
Block #12345678
├── ヘッダー
│   ├── previousHash   : 前ブロックのハッシュ（チェーンを形成する鍵）
│   ├── timestamp      : ブロック生成時刻
│   ├── nonce          : マイニング時に使った値（PoW時代の名残）
│   ├── stateRoot      : 全アカウント残高のスナップショット（Merkleツリー）
│   ├── transactionsRoot: このブロック内のTX一覧
│   └── receiptsRoot   : 各TXの実行結果
└── トランザクション一覧
    ├── TX: 0xabc... → from: Alice, to: Bob, value: 1ETH
    ├── TX: 0xdef... → JPYC transfer: Alice → Charlie, 1000JPYC
    └── ...（1ブロックに数百〜数千TX）
```

### なぜデータを改ざんできないのか

改ざんが実質不可能な理由は「ハッシュ関数の特性」にあります。

**ハッシュ関数とは:**
- 任意のデータを一定長の固定値（ハッシュ）に変換する
- 入力が1ビットでも変わると出力が完全に別の値になる
- 出力から入力を逆算することが計算上不可能

**改ざんが検知される仕組み:**
```
Block #100 を改ざんしようとすると...
  → Block #100 のハッシュが変わる
  → Block #101 が持つ「前ブロックのハッシュ」と不一致になる
  → Block #101 以降のハッシュがすべて狂う
  → ネットワーク上の他ノードと一致しなくなる
  → 改ざんが即座に検知される
```

世界中に存在する数千ノードのデータと常に照合されるため、
一部のノードのデータを書き換えても無意味です。

## 2. Ethereum とは何か

### Bitcoin との違い

| 項目 | Bitcoin | Ethereum |
|---|---|---|
| 目的 | デジタル通貨（価値の移転）| プログラマブルな分散コンピュータ |
| スクリプト | 限定的（送金条件のみ）| Turing 完全なプログラム実行 |
| スマートコントラクト | 不可 | 可能 |
| ネイティブ通貨 | BTC | ETH（ガス代として使用）|
| コンセンサス | PoW (Proof of Work) | PoS (Proof of Stake, 2022年移行後）|

Bitcoin は「送金」に特化しています。
Ethereum は「**EVM（Ethereum Virtual Machine）**」という仮想マシンを持ち、
任意のプログラム（スマートコントラクト）をチェーン上で実行できます。

### なぜスマートコントラクトが動くのか（EVM とは）

**EVM（Ethereum Virtual Machine）**は、Ethereum ネットワーク上のすべてのノードが持つ仮想マシンです。

```
スマートコントラクトのデプロイ:
  開発者がコード（Solidity）を書く
  → EVM バイトコードにコンパイル
  → TX としてチェーンに送信
  → 全ノードがそのコードを保存
  → コントラクトアドレスが割り当てられる

コントラクトの実行:
  ユーザーがコントラクトアドレスに TX を送信
  → 全ノードが同じコードを同じ入力で実行
  → 全ノードで同じ結果が得られる（決定論的実行）
  → ブロックに記録される
```

「全ノードが同じプログラムを実行して同じ結果になる」ことが EVM の本質です。
これにより「**信頼できる第三者なしに契約を自動執行できる**」という特性が生まれます。

JPYC もスマートコントラクトとして Polygon チェーン上に存在します。
`transfer()` を呼ぶと、全ノードがその残高更新を検証・記録します。

### ガス（Gas）とは何か・なぜ必要か

**ガス**は Ethereum の「計算費用」の単位です。

```
TX の実行コスト = ガス量 × ガス単価（Gwei）

例: 単純な ETH 送金
  ガス量: 21,000 gas（固定）
  ガス単価: 20 Gwei = 0.000000020 ETH
  合計: 21,000 × 0.000000020 = 0.00042 ETH ≈ 数円〜数十円

例: ERC-20 transfer（JPYC送金など）
  ガス量: 約 50,000〜65,000 gas（コントラクト実行分が追加）
  ガス単価: ネットワーク混雑度で変動
```

**なぜ必要か:**
1. **スパム防止:** 無限ループするコードをコストなしに実行できると、ネットワークが詰まる
2. **マイナー/バリデーターへの報酬:** ブロックを生成する参加者への対価
3. **計算リソースの公平な分配:** 計算量に応じた公平なコスト負担

本プロジェクトでは **EIP-2612 Permit** を使って、
ユーザーのガス代負担をバックエンドが肩代わりする設計を Phase 5 で実装します。

## 3. ノードとは何か

### フルノード・ライトノード・アーカイブノードの違い

| 種類 | 保持データ | 用途 |
|---|---|---|
| **フルノード** | 最近の状態 + 全ブロックヘッダー | 送金・コントラクト実行の検証 |
| **ライトノード** | ブロックヘッダーのみ | モバイルウォレットなど軽量クライアント |
| **アーカイブノード** | 全歴史状態（全ブロックの全残高）| 過去の残高照会・分析用 |

本プロジェクトで使う `eth_getBalance` や `eth_getLogs` はフルノードで対応可能です。
ただし過去のブロック（数千ブロック以上前）のデータを大量に取得するにはアーカイブノードが必要になることがあります。

### Infura / Alchemy はどういうサービスか

自前でフルノードを立てるには:
- 500GB〜2TB以上のストレージ
- 常時稼働するサーバー
- 同期に数日かかる初期セットアップ

**Infura / Alchemy** は、この「ノードをクラウドで運用するサービス」です。

```
私たちのアプリ
  ↓ HTTPS リクエスト (JSON-RPC)
Infura / Alchemy のゲートウェイ
  ↓
世界中のフルノード / アーカイブノード
  ↓
Ethereum ネットワーク
```

**無料枠の目安:**
- Infura: 1日 100,000 リクエスト（月 3,000,000）
- Alchemy: 月 300,000,000 コンピュートユニット

本プロジェクトでは開発・テスト目的のため、無料枠で十分に対応できます。

### JSON-RPC とは何か（eth_blockNumber などの意味）

**JSON-RPC** は、JSON 形式でリモートプロシージャコール（RPC）を行うプロトコルです。
Ethereum ノードはこのインターフェースで外部からの操作を受け付けます。

```json
// リクエスト
{
  "jsonrpc": "2.0",
  "method": "eth_blockNumber",
  "params": [],
  "id": 1
}

// レスポンス
{
  "jsonrpc": "2.0",
  "result": "0xbc614e",  // 16進数: 12345678 ブロック
  "id": 1
}
```

主要な JSON-RPC メソッド:

| メソッド | 説明 |
|---|---|
| `eth_blockNumber` | 最新ブロック番号を取得 |
| `eth_getBalance` | アドレスの ETH 残高を取得 |
| `eth_call` | 読み取り専用のコントラクト呼び出し（ガス不要）|
| `eth_sendRawTransaction` | 署名済みTXをブロードキャスト |
| `eth_getLogs` | イベントログを取得（Transfer イベント監視に使用）|
| `eth_chainId` | 接続中のチェーン ID を取得 |

**Web3j** はこれらの JSON-RPC 呼び出しを Java のメソッドとしてラップしてくれるライブラリです。

```java
// Web3j を使うと:
BigInteger blockNumber = web3j.ethBlockNumber().send().getBlockNumber();

// 内部では以下の JSON-RPC が実行される:
// POST /  {"method": "eth_blockNumber", ...}
```

## 4. メインネット vs テストネット

### Sepolia とは何か

**Sepolia** は Ethereum の公式テストネットワーク（テストチェーン）です。

```
Mainnet (chainId: 1)
  → 本物の ETH・本物のトークンが動く
  → 操作ミスは取り返しがつかない

Sepolia (chainId: 11155111)
  → テスト用の ETH（価値ゼロ）を使う
  → 何度でも失敗OK・本番と同じ動きをする
  → Faucet でテスト ETH を無料取得できる
```

以前は **Ropsten・Rinkeby・Goerli** などが使われていましたが、現在は **Sepolia が推奨**テストネットです。

### テストネットの ETH はどこで手に入るか（Faucet）

**Faucet（フォーセット）** = テスト用 ETH を無料配布するサービス

- **Alchemy Sepolia Faucet**: [https://sepoliafaucet.com](https://sepoliafaucet.com)
- **Infura Faucet**: Infura ダッシュボード内から取得可能
- 1回のリクエストで 0.5 ETH 程度（テスト用）

### なぜテストネットで開発するのか

| 理由 | 説明 |
|---|---|
| **コスト** | テストネットの ETH は無料 → 何度でも試せる |
| **安全性** | バグがあっても本物の資産に影響しない |
| **速度** | 本番同等の挙動をより速く確認できる |
| **リセット可能** | テストネット上のデータは定期的にリセットされる場合がある |

## 5. このプロジェクトとの関係

### JPYC の送金がブロックチェーン上でどう記録されるか

JPYC は Polygon（Ethereum 互換チェーン）上の **ERC-20 コントラクト**として存在します。

```
ユーザーが JPYC を送金するとき（大まかな流れ）:

1. ユーザーのウォレット（HashPort）が TX を作成
   {
     to: "0x431D5dfF...BDB",  ← JPYC コントラクトアドレス
     data: "transfer(0xRecipient, 1000000000000000000)"  ← 1 JPYC（18 decimals）
   }

2. 秘密鍵で TX に署名
   → 誰が送ったかを暗号学的に証明

3. TX をネットワークにブロードキャスト
   → バリデーターがピックアップ

4. バリデーターが TX を実行
   → JPYC コントラクトの transfer() を実行
   → 残高マッピングを更新: balances[from] -= amount; balances[to] += amount;
   → Transfer イベントを発火: Transfer(from, to, amount)

5. ブロックに含まれて確定
   → TX ハッシュが発行される（例: 0xabc...def）
   → 全ノードがこの状態変化を保持

6. 本プロジェクトのバックエンドがイベントを検知（Phase 4 で実装）
   → eth_getLogs で Transfer イベントを検索
   → receiverAddress が一致 → PaymentOrder を CONFIRMED に更新
```

### まとめ: なぜブロックチェーンを使うのか

```
従来の決済:
  ユーザー → 銀行API → 振込処理 → 受取銀行 → 相手に着金
  （中間業者・手数料・数時間〜翌営業日）

ブロックチェーン決済（JPYC）:
  ユーザー → コントラクト直接呼び出し → 数秒〜数十秒で着金
  （中間業者なし・少額手数料・24/365稼働）
```

本プロジェクトは「ブロックチェーンの送金を検知してビジネスロジックに繋ぐ」バックエンドです。
チェーン上の Transfer イベントをバックエンドが監視し、決済の完了を確認します。
