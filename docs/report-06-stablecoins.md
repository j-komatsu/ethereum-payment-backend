# レポート06 — ステーブルコイン：設計思想・リスク・実装との関係

## はじめに

Phase 3〜5 を通じて JPYC・USDC・USDT・DAI を扱ってきた。
本レポートでは「なぜステーブルコインは価格が安定するのか」「各コインの設計はどう違うのか」「リスクは何か」を整理する。

---

## 1. そもそも「ステーブルコイン」とは何か

ビットコインや Ethereum の価格は激しく変動する（1日で±20%も珍しくない）。これでは「決済手段」として使えない。

```
2021年11月: ETH = 約50万円
2022年6月:  ETH = 約15万円（7ヶ月で70%下落）
```

ステーブルコインは**法定通貨（円・ドル）などにペッグ（連動）させた価格安定型トークン**。

```
1 USDC ≈ 1 USD（常に）
1 JPYC ≈ 1 JPY（常に）
```

---

## 2. ペッグの維持メカニズム（種類別）

ステーブルコインには3種類の設計があり、「どうやって$1や1円を維持するか」の仕組みが根本的に異なる。

### 種類1: 法定通貨担保型（Fiat-backed）

**代表: USDC、USDT、JPYC**

```
発行体が銀行口座に $100 を預ける
    ↓
100 USDC を発行（Ethereum 上に）
    ↓
ユーザーが 100 USDC を返すと $100 を払い戻す
```

仕組みとして最もシンプル。1トークンに対して1単位の法定通貨が実際に存在する（はず）。

| 特徴 | 内容 |
|---|---|
| 安定性 | 高い（現金で裏付け） |
| 信頼の前提 | 発行体が本当に現金を持っているか |
| 中央集権 | 強い（発行体が凍結・没収できる） |
| 規制 | 各国の金融規制に服する |

### 種類2: 暗号資産担保型（Crypto-backed）

**代表: DAI（MakerDAO）**

```
ユーザーが ETH を担保に預ける（例: 150ドル分）
    ↓
100 DAI を発行（担保率 150%）
    ↓
ETH の価格が下がると自動清算
```

「分散型」の設計。発行体や銀行は存在せず、スマートコントラクトが自動で管理する。

```
なぜ 150% の担保が必要か？
    ETH が 30% 下落 → 担保 = 105ドル → まだ 100 DAI を裏付けられる
    ETH が 50% 下落 → 担保 = 75ドル → 清算発動（ポジションを強制決済）
```

| 特徴 | 内容 |
|---|---|
| 安定性 | 中（ETH の暴落時に圧力がかかる） |
| 信頼の前提 | スマートコントラクトのコードが正しいか |
| 中央集権 | 弱い（コードが自動管理） |
| 規制 | 回避しやすいが法整備が追いついていない |

### 種類3: アルゴリズム型（Algorithmic）

**代表: UST（Terra/Luna）— 2022年に崩壊した失敗例**

担保を持たず、アルゴリズムとガバナンストークンの供給量操作でペッグを維持しようとした。

```
UST が $0.99 に下落
    ↓
UST を焼却 → LUNA を発行（供給増加で UST の希少性を回復）
    ↓
LUNA の価格が下がる → 更に UST が売られる → デススパイラル
```

2022年5月: UST が $1 → $0.0001 に暴落。数十兆円が蒸発。  
→ **担保のないペッグは信頼が崩れると止められない**という教訓。

## 3. 各コインの詳細

### USDC（USD Coin）

- **発行体:** Circle（米国企業）
- **担保:** 米国財務省短期証券 + 現金
- **監査:** 毎月 Grant Thornton（会計事務所）が証明書を発行
- **特徴:** 規制準拠を重視。疑わしいアドレスをブラックリスト化できる
- **EIP-2612:** 対応（version="2"）

```java
USDC("0xA0b86991c6218b36c1d19D4a2e9Eb0cE3606eB48", 6, 1, "USD Coin", "2", true)
```

### USDT（Tether）

- **発行体:** Tether Limited（英領ヴァージン諸島）
- **担保:** 長らく「100% USD 裏付け」と主張するも、実際は商業手形・暗号資産なども混在
- **流動性:** 時価総額・取引量ともに世界最大
- **リスク:** 透明性が低い。2021年に米 CFTC から罰金（担保不足の虚偽申告）
- **EIP-2612:** **非対応**（Ethereum Mainnet の USDT は permit 関数を持たない）

```java
USDT("0xdAC17F958D2ee523a2206206994597C13D831ec7", 6, 1, "Tether USD", "1", false) // not EIP-2612
```

> このプロジェクトでは `permitSupported=false` として入口で拒否している。

### DAI

- **発行体:** MakerDAO（DAO = 分散自律組織）
- **担保:** ETH・WBTC・USDC など多様な暗号資産（過担保）
- **特徴:** 完全分散型。コードがルール。管理者が存在しない
- **EIP-2612:** 対応（version="1"）

2023年以降は USDC を大量担保に取り込んだため「完全分散型」ではなくなりつつある。

### JPYC（JPYコイン）

- **発行体:** JPYC Inc.（東京・日本企業）
- **担保:** 日本円（前払式支払手段 / 電子決済手段）
- **法的根拠:** 資金決済法に基づく「前払式支払手段」→ 2024年から「電子決済手段」に移行
- **チェーン:** Polygon Mainnet（ガス代が POL で安い）
- **EIP-2612:** 対応（version="1"）
- **特徴:** 日本初の法令準拠型円建てステーブルコイン

```
電子決済手段としての規制:
  → 年次監査が義務
  → 準備金の 100% 保全が必要
  → 金融庁への届出
```

## 4. なぜ decimals が違うのか（設計の思想）

| トークン | decimals | 最小単位 |
|---|---|---|
| USDC | 6 | $0.000001（1マイクロドル） |
| USDT | 6 | $0.000001 |
| DAI | 18 | $0.000000000000000001 |
| JPYC | 18 | ¥0.000000000000000001 |

**USDC/USDT が 6 の理由:** 「ドルの決済精度は $0.01 で十分、余裕を持って 6 桁にした」という Circle/Tether の設計判断。

**DAI/JPYC が 18 の理由:** ETH と単位を合わせることでコントラクト間計算（DeFi）がシンプルになる。18桁は EVM の uint256 の最大精度に合わせた設計。

## 5. ペッグが崩れるとき

| 要因 | 例 |
|---|---|
| 担保不足が発覚 | USDT の準備金疑惑（市場の信頼喪失） |
| アルゴリズムの前提崩壊 | UST/LUNA のデススパイラル |
| 発行体の倒産 | Silicon Valley Bank 破綻 → USDC 一時 $0.87 に（2023年） |
| 規制当局の介入 | 凍結・没収（USDC には実際にその機能がある） |

> **2023年3月の USDC ショック:**  
> Circle が SVB に預金していた 33億ドルが引き出せなくなるリスクから  
> USDC が一時 $0.87 まで下落。72時間後に FDIC が預金保護を表明して回復。

## 6. このプロジェクトとの関係

### なぜ JPYC を選んだか

```
円建て → 日本のユーザーにとって価格変動リスクなし
Polygon → ガス代が安い（ETH の 1/100 以下）
EIP-2612 対応 → ガスレス Permit 払いが使える
```

### Permit がガスレスになる理由

ERC-20 の通常フロー（approve + transferFrom）では**ユーザー自身がガス代（POL）を持つ必要がある**。

```
通常フロー:
  ユーザー → approve() TX → ガス代必要（POL）
  スペンダー → transferFrom() TX → ガス代必要（POL）
```

EIP-2612 Permit フロー:

```
Permit フロー:
  ユーザー → 署名（オフチェーン） → ガス代ゼロ
  スペンダー → permit() TX + transferFrom() TX → ガス代はスペンダーが負担（POL）
```

ユーザーが POL を持っていなくても支払いができる → **UX が大幅に向上**。

## まとめ

| 観点 | USDC | USDT | DAI | JPYC |
|---|---|---|---|---|
| 担保 | USD（透明） | USD他（不透明） | 暗号資産（過担保） | 円（法的保全） |
| 分散度 | 中央集権 | 中央集権 | 分散 | 中央集権 |
| 規制準拠 | 高 | 低 | なし | 高（日本法） |
| EIP-2612 | ✅ | ❌ | ✅ | ✅ |
| このプロジェクト | 対応 | 非対応（permit不可）| 対応 | **メイン** |

ステーブルコインの「安定性」は**信頼に基づく**。その信頼は担保（何が裏付けか）と発行体（誰が管理するか）の透明性から来る。アルゴリズム型の失敗が示すように、**経済的なインセンティブだけで信頼は維持できない**。
